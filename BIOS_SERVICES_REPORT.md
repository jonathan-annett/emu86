## Architecture ‚Äî no custom opcodes, trap-based dispatch

The data flow for a guest `INT N`:

```
guest CS:IP  ‚îÄ‚îÄ‚îÄ 0xCD N ‚îÄ‚îÄ‚ñ∫  CPU8086.step()
                              ‚îú‚îÄ INT N (real silicon): push FLAGS, push CS,
                              ‚îÇ  push IP, IF‚Üê0, far-jump through IVT[N]
                              ‚îî‚îÄ CS:IP ‚Üê IVT[N] = F000:1000+N

next step():    CS:IP = F000:1000+N
                ‚îú‚îÄ TrapRegistry.get(0xF1000+N)  ‚Üí JS handler
                ‚îÇ   handler runs: read inputs from cpu.regs, do the work
                ‚îÇ   via Console / Disk / HostClock / BDA, write outputs
                ‚îÇ   back to cpu.regs and the pushed FLAGS word
                ‚îî‚îÄ CPU then fetches the byte at F000:1000+N (= 0xCF, IRET)
                   IRET pops IP, CS, FLAGS ‚Äî control returns to the
                   instruction after `INT N` in the guest, with handler-
                   modified registers and CF.
```

Two `cpu.step()` calls per `INT N`: one for `INT`, one for the trap
fire + IRET. The CPU saw exactly two real-silicon instructions. The trap
registry is a `Map<linear, TrapHandler>` consulted at the top of every
step ‚Äî empty-registry cost is one optional `Map.get` (effectively free
when the map is unset, which is how the SST corpus runs).

The init code in the BIOS ROM is hand-encoded 8086 (~111 bytes). It is
real machine code ‚Äî the CPU executes it like any other program. It builds
the IVT, overrides IVT[0x1E] for the diskette parameter table, populates
the BDA, sets up SS:SP, and `INT 19h`s into the boot sector. No JS
involvement until the trap fires for `INT 19h` itself.

## HostClock

`src/host-clock/host-clock.ts` defines:

```ts
interface HostTime { seconds, minutes, hours, dayOfMonth, month,
                     year /* 1900-based */, dayOfWeek, dayOfYear,
                     dst, milliseconds }

interface HostClock { now(): HostTime }
```

Two implementations:

- **`NodeHostClock`** ‚Äî production. Reads `new Date()` and projects it
  into `HostTime`. Used by default in the Machine constructor.
- **`InMemoryHostClock`** ‚Äî tests. Default initial time is
  2026-01-01 00:00:00 (a Thursday ‚Äî `dayOfWeek = 4`, matching the
  current project date). Methods:
  - `setTime(t)` overwrites the stored value verbatim.
  - `advance(ms)` propagates carries through ms ‚Üí s ‚Üí m ‚Üí h ‚Üí days,
    including day-of-week and day-of-year. `advance(86_400_000)` cleanly
    rolls midnight.
  - `now()` returns a defensive copy (mutating the result does not leak
    in).

Both are covered by `tests/unit/host-clock.test.ts`.

## BIOS ROM layout

Generated by `buildBiosRom()` in `src/bios/bios-rom.ts`. Pure function ‚Äî
deterministic byte output, verified by `tests/unit/bios-rom.test.ts`.

| Linear range          | Offset within ROM | Contents                                                                                       |
| --------------------- | ----------------: | ---------------------------------------------------------------------------------------------- |
| `0xF0000 .. 0xF00FF`  |          `0x0000` | Filled with `0xFF` (open-bus / blank EPROM convention).                                        |
| `0xF0100 .. 0xF016E`  |          `0x0100` | **BIOS init code** ‚Äî ~111 bytes, hand-encoded. Build IVT, override IVT[0x1E], setup BDA, boot. |
| `0xF016F .. 0xF0FFF`  |                 ‚Äî | `0xFF` (slack between init and the trap table).                                                |
| `0xF1000 .. 0xF10FF`  |          `0x1000` | **Trap stub table** ‚Äî 256 single-byte `IRET` (`0xCF`) stubs, one per IVT entry.                |
| `0xF1100 .. 0xF1FFF`  |                 ‚Äî | `0xFF`.                                                                                        |
| `0xF2000 .. 0xF200A`  |          `0x2000` | **Diskette parameter table** ‚Äî 11 bytes, typical 1.44 MB floppy.                               |
| `0xF200B .. 0xFFFEF`  |                 ‚Äî | `0xFF`.                                                                                        |
| `0xFFFF0 .. 0xFFFF4`  |          `0xFFF0` | **Reset vector** ‚Äî `EA 00 01 00 F0` = `JMP F000:0100`.                                         |
| `0xFFFF5 .. 0xFFFFF`  |                 ‚Äî | `0xFF`.                                                                                        |

`buildBiosRom()` returns `{ bytes, baseLinear: 0xF0000, layout }`. The
layout exposes:

- `trapAddresses[N]` = `0xF1000 + N` for every `N ‚àà [0, 256)`.
- `resetVector = 0xFFFF0`, `initEntry = 0xF0100`,
  `disketteParamTable = 0xF2000`.
- `initCodeLength` so tests can confirm we're under the ~200-byte budget
  the brief calls out.

The Machine calls `memory.loadROM(0xF0000, bytes)` ‚Äî the existing
`PagedMemory.loadROM` mechanism marks the F-region pages read-only.

## Service-by-service

All handlers live in `src/bios/bios-services.ts` and are wired into
their trap addresses by `registerBiosHandlers(registry, layout, ctx)`.

| INT  | AH subfunctions                        | Notes                                                                                                        |
| ---- | -------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| 08h  | (hardware timer)                       | Increments `BDA.CLK_DTIMER` (32-bit), writes EOI (`0x20`) to PIC port `0x20`. Does **not** chain to INT 1Ch. |
| 10h  | 00, 02, 03, 06, 07, 08, 09, 0E, 0F     | Cursor tracked in BDA. Writes go to `Console.writeChar`. Scroll (06/07 AL=0) emits a newline.                |
| 11h  | (no AH dispatch)                       | Returns `AX = BDA.EQUIPMENT` or `0x0021` default.                                                            |
| 12h  | (no AH dispatch)                       | Returns `AX = BDA.MEMORY_SIZE_KB` or `0x0280` (640 KiB) default.                                             |
| 13h  | 00, 01, 02, 03, 04, 05, 08, 15, 16     | CHS‚ÜîLBA in handler; routes to `Disk.readSector`/`writeSector`. Status stored in `BDA.DISK_LASTSTATUS`.       |
| 16h  | 00, 01, 02                             | Drains `Console.hasInput()` into BDA kb-buffer at each call. AH=00 returns `AX=0` on empty (see below).      |
| 19h  | (bootstrap)                            | Reads sector 0 to `0:7C00`, sets `DL=0`, modifies pushed CS:IP so the IRET lands at `0:7C00`.                |
| 1Ah  | 00, 02, 04                             | Reads `HostClock.now()`. AH=00 returns ticks at 65536/3600 Hz; AH=02/04 return BCD time/date.                |

Vectors **not** registered: every other vector (0x00‚Äì0x07 except 8,
0x14, 0x15, 0x17, 0x1B‚Äì0x1F, etc.) keeps its bare `IRET` stub. That is
exactly the right behaviour for stubs ‚Äî `IRET` returns immediately with
flags / registers untouched, which is the correct no-op for, e.g., `INT
1Bh` (Ctrl-Break handler) or unused IRQs. INT 14h (serial) and INT 17h
(printer) we stub now; if Phase 3 needs them we add real handlers.

Unimplemented AH subfunctions in INT 10h / 13h / 16h / 1Ah call
`ctx.warn(...)` so test harnesses can detect them; the call still
returns gracefully (typically with an error status for INT 13h, no-op
elsewhere).

## Pushed-flags-on-stack mechanism

When the CPU executes `INT N` it pushes (high ‚Üí low addresses):
`FLAGS, CS, IP`. After the push, `SS:SP` points at IP. The trap stub
runs *before* the IRET fires, so the handler can rewrite what IRET will
pop:

```
SS:SP+0   IP    ‚óÑ‚îÄ‚îÄ IRET pops first
SS:SP+2   CS
SS:SP+4   FLAGS
```

`bios-services.ts` exposes three small helpers:

- `setReturnCF(cpu, value)` ‚Äî flips bit 0 (CF) in the FLAGS word at
  `SS:SP+4`.
- `setReturnZF(cpu, value)` ‚Äî flips bit 6 (ZF). Used by `INT 16h AH=01h`.
- `setReturnCsIp(cpu, segment, offset)` ‚Äî overwrites the IP and CS words
  at `SS:SP+0` and `SS:SP+2`. Used by `INT 19h` to "jump" the caller into
  the boot sector at `0:7C00`.

These helpers are the only place handlers touch the stack ‚Äî they keep
the layout knowledge in one place. The IRET that follows pops whatever
the handler left there.

## Keyboard input path

We chose the **"even simpler" direct-injection path** that the brief
recommends:

- The Machine does **not** wire Console input to PIC IRQ 1 / INT 9. The
  trap stub at `F000:1009` is just `IRET` and is never registered with a
  JS handler.
- Instead, the `INT 16h` handler itself drains `Console.hasInput()` /
  `Console.readChar()` into the BDA keyboard buffer at the top of every
  call (`drainConsoleIntoKbBuffer`).

Why: in a headless emulator the BDA is the source of truth. There is no
real keyboard controller on port 0x60, no scancode-translation step
worth modelling, and no benefit to the extra IRQ-9 round trip ‚Äî the
guest can only observe keyboard state via `INT 16h` anyway. Async
keypress timing in real hardware doesn't matter when reads are entirely
on-demand.

If Phase 3 surfaces ELKS code that polls IRQ 1 directly (interrupt-
driven keystrokes, e.g. terminal `getch` via `INT 9` hooks), we can wire
the Console-input ‚Üí BDA + `pic.assertIRQ(1)` path then. The handler
contract for INT 16h is unchanged ‚Äî it would still be the consumer.

## INT 16h AH=00h "blocking" choice

Per the brief's "for our v0, do the simpler thing": when the BDA kb
buffer is empty, AH=00h returns **`AX = 0`** with CF=0 and **does not
block**. This is documented in `int16Handler` and is the single
known-divergence point between our BIOS and a real PC BIOS that loops
HLT/INT-9 until a key arrives.

Rationale: blocking would require either (a) a synchronous spin in the
handler that reads from Console (only works for callers who pre-staged
input), or (b) decrementing the pushed IP back to point at the `INT 16h`
byte and HLTing ‚Äî workable but invasive and not needed for any v0 test.
The brief explicitly accepts this v0 behaviour and reserves the question
for Phase 3 ELKS triage.

AH=01h (status check) is fully implemented with ZF semantics: ZF=1 on
empty, ZF=0 + AX = next key (without popping) when present. Code that
checks status before reading will behave correctly. AH=02h returns the
BDA shift-flags byte.

## Things Phase 3 will need

- **Disk attachment.** The Machine accepts an optional `disk: Disk` in
  config; pass an `InMemoryDisk` (tests) or a future `NodeFileDisk`
  loaded from an ELKS image. With no disk, `INT 19h` sets CF and falls
  through to the safety-net HLT ‚Äî observable, but not a boot.
- **No INT 1Ch chaining.** INT 8 increments the tick counter and EOIs;
  it does **not** dispatch INT 1Ch. If ELKS hangs a user-time-tick hook
  off INT 1Ch (DOS-style cooperative multitasking), Phase 3 will need to
  add a re-entrant call from the int8 handler. The reason we skipped: a
  trap handler can't synchronously re-enter `cpu.step()` to issue
  another `INT N`; it would need to manipulate FLAGS/CS/IP on the stack
  and arrange a controlled "fall-through" ‚Äî workable but non-trivial.
- **Multi-drive support.** INT 13h accepts `DL = 0x00` (floppy 0) or
  `DL = 0x80` (HD 0); other drive numbers return BAD_COMMAND. ELKS may
  probe drives; if so, surface and add.
- **No INT 9 / IRQ 1.** As above. If ELKS keyboard ISR registration is
  observed, wire it.
- **Video mode tracking is shallow.** AH=00h stashes the requested mode
  in BDA; AH=0Fh reports it back. We do not model CGA/EGA palettes,
  scroll regions beyond the AL=0 full-screen case, or screen memory.
- **`INT 13h AH=02h` reads only the first 256 sectors per call** (AL is
  a byte). This matches the real BIOS contract and is not a limitation;
  noting because LBA-style reads (AH=42h, EXT13H) are not implemented.
